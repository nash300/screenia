# QA Customer Fixtures

The Screenia QA dataset is generated by `scripts/qa-customers.mjs`. It creates
100 fictional customers in 20 lifecycle groups with five variations per group.
All customer emails use the reserved `example.test` domain, and every record is
marked with `SCREENIA_QA_100_V1` so it can be removed without touching real
customers.

## Commands

- `npm run qa:customers:seed` replaces the existing QA batch and verifies it.
- `npm run qa:customers:verify` checks fixture counts without changing data.
- `npm run qa:customers:clean` removes only this QA batch.

The seeder refuses to run when non-QA customers exist. It never sends email and
does not create Stripe objects. Ten confirmed Supabase Auth test accounts are
created without email delivery. The first account is
`qa.customer.041@example.test` with password `ScreeniaQa12345`.

## Scenario Groups

1. New website request, including incomplete contact details.
2. Quote prepared but not sent.
3. Quote and setup link sent, including an expired link.
4. Profile completed while Checkout remains unfinished.
5. First payment completed and material collection pending.
6. Customer material remains incomplete.
7. Material received but no device allocated.
8. Device allocated without playlist content.
9. Active service with a playable display and customer account.
10. Active service inside the 21-day trial.
11. Failed recurring payment with suspended access.
12. Temporarily paused subscription.
13. Cancellation scheduled for the current period end.
14. Cancelled and closed customer.
15. Full refund before layout work starts.
16. Partial refund request after layout work starts.
17. Payment dispute under review.
18. Device prepared and ready to ship.
19. Shipped order with tracking evidence.
20. Temporary discount with support cases in multiple states.

Each group alternates Standard FHD and Premium 4K. Variations cover one to four
matching devices and a mixed two-device order containing one of each package.
Stored first-payment totals follow Screenia's inclusive-moms model: one setup
fee plus each device and each shipping charge, less any device discount.

## Provider Boundary

The database fixtures intentionally have no Stripe customer, Checkout,
PaymentIntent, invoice, or subscription IDs. They test admin presentation,
filtering, workflow queues, local audit history, account linking, and database
relationships. Actual payment, webhook, billing portal, refund, dispute, and
invoice behavior must still be tested using controlled Stripe test-mode events.
The discount fixture uses an explicit `qa_no_stripe_...` reference only because
the adjustment table requires a subscription reference; provider operations on
that fixture are expected to fail safely.
